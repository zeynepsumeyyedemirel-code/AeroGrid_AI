## Project Overview

Field technicians servicing wind turbines and solar installations often work in remote, off-grid locations — mountain ridges, offshore platforms, desert solar farms — where cloud connectivity is unreliable or nonexistent. Large Language Models are powerful, but they are not useful in the field if they require an internet connection, and they can hallucinate when asked about equipment-specific fault codes or safety procedures.

AeroGrid AI solves this by running the entire pipeline **on-device**: technical documentation is embedded and indexed locally, and answers are generated by a local LLM (Phi-3 via Ollama) — no data ever leaves the technician's laptop.

> **Key idea:** Keep the manuals local, retrieve only the most relevant fault-code or safety context, and generate grounded answers without needing a signal bar.

## Features

- **Fully offline** — no cloud dependency, no API keys, works with zero internet access
- **Retrieval-Augmented Generation (RAG)** — answers are grounded in the technician's own manuals, not the model's general knowledge
- **Local semantic search** — Sentence-Transformers embeddings + cosine similarity, no external vector DB required
- **Source-verified answers** — responses reference the fault code / document section they came from, so technicians can double-check against the manual
- **Safety-guarded prompting** — high-voltage and lockout/tagout (LOTO) context is prioritized in the system prompt
- **Simple Streamlit dashboard** — usable on a laptop in the field with no build step

## System Architecture & RAG Pipeline

```
┌──────────────────────────┐        ┌──────────────────────────┐
│ Technical Docs (.txt)    │        │ User Query (Field)       │
│ Wind / Solar / Safety    │        │ "E-301 fault code"       │
└────────────┬──────────────┘        └────────────┬──────────────┘
             │                                    │
             ▼                                    ▼
┌──────────────────────────┐        ┌──────────────────────────┐
│ Sentence-Transformers     │        │ Query Vector Encoding    │
│ (all-MiniLM-L6-v2)       │        └────────────┬──────────────┘
└────────────┬──────────────┘                     │
             │                                    │
             ▼                                    ▼
┌──────────────────────────┐        ┌──────────────────────────┐
│ Local Vector Index        ├───────►│ Cosine Similarity Match  │
│ Matrix (In-Memory)        │        │ (Top-K Context Chunks)   │
└──────────────────────────┘        └────────────┬──────────────┘
                                                  │
                                                  ▼
┌──────────────────────────┐        ┌──────────────────────────┐
│ Safety & Context Guarded  ├───────►│ Ollama (Microsoft Phi-3) │
│ System Prompt             │        │ On-Device LLM Inference  │
└──────────────────────────┘        └────────────┬──────────────┘
                                                  │
                                                  ▼
                                       ┌──────────────────────────┐
                                       │ Field Guidance & Source  │
                                       │ Verification Context UI │
                                       └──────────────────────────┘
```

**How a query flows:**

1. The technician types a question (e.g. *"E-301 fault code"*) into the Streamlit dashboard.
2. The query is embedded with the same Sentence-Transformers model used to index the documents.
3. The in-memory vector index ranks all document chunks by cosine similarity and returns the top-K most relevant ones.
4. Those chunks are inserted into a safety-guarded system prompt, along with the user's question.
5. Ollama runs Phi-3 locally to generate a grounded answer.
6. The dashboard displays the answer alongside the source document/section it was derived from.

## Technology Stack

| Technology | Purpose |
|---|---|
| Streamlit | Local web dashboard / field UI |
| Sentence-Transformers (`all-MiniLM-L6-v2`) | Converts documents and queries into vector embeddings |
| Cosine similarity (in-memory matrix) | Semantic retrieval of the top-K relevant chunks |
| Ollama + Phi-3 | On-device LLM inference, no cloud calls |
| Python | Application logic |

## Local Setup & Execution

### Prerequisites (Ollama & Model)

```bash
ollama pull phi3
```

### Clone & Environment Setup

```bash
git clone https://github.com/zeynepsumeyyedemirel-code/AeroGrid_AI.git
cd AeroGrid_AI

python3 -m venv AeroGrid_venv
source AeroGrid_venv/bin/activate  # on Windows: AeroGrid_venv\Scripts\activate

pip install streamlit sentence-transformers torch requests
```

### Launch Local Field Assistant

```bash
streamlit run dashboard.py
```

The dashboard will open at `http://localhost:8501`.

## Document Knowledge Base Structure

The local vector engine indexes 16 technical documents across three categories in the `docs/` directory. Each fault code or procedure lives in its own file, so retrieval returns focused, single-topic chunks rather than large mixed-topic blocks.

**`docs/wind_turbine/`** — one file per fault code, covering symptoms, likely causes, diagnostic steps, resolution, and safety notes:
- `E-101-pitch-control.txt` — Pitch control system fault (hydraulic/electrical)
- `E-150-anemometer-fault.txt` — Anemometer / wind vane fault
- `E-220-brake-wear.txt` — Mechanical brake pad wear limit
- `E-301-stator-overheating.txt` — Generator stator overheating
- `E-404-yaw-jam.txt` — Yaw system jam
- `E-501-gearbox-vibration.txt` — Gearbox excessive vibration

**`docs/solar_panel/`** — one file per fault code:
- `F-45-inverter-overtemperature.txt` — Inverter over-temperature
- `F-60-string-isolation-fault.txt` — String isolation / combiner fuse fault
- `F-80-grid-overvoltage.txt` — Grid overvoltage trip
- `F-102-ground-fault.txt` — DC array ground fault
- `F-133-mppt-tracking-error.txt` — MPPT tracking error

**`docs/safety/`** — one file per procedure:
- `loto-8-step-procedure.txt` — Lockout/Tagout isolation procedure
- `arc-flash-boundaries.txt` — Arc flash boundaries & PPE categories
- `hv-ppe-requirements.txt` — High-voltage PPE requirements
- `confined-space-entry.txt` — Confined space entry (tower base/foundation)
- `fall-protection-height.txt` — Fall protection for work at height

> Splitting each fault code/procedure into its own file (instead of three large combined files) improves retrieval precision: a query about "E-501 gearbox vibration" now matches one focused document instead of a chunk that also contains unrelated pitch-control or yaw content.

> ⚠️ **Important:** the documents are now organized into `docs/wind_turbine/`, `docs/solar_panel/`, and `docs/safety/` subfolders instead of being flat files directly in `docs/`. If `retriever.py` currently loads files with a pattern like `glob("docs/*.txt")`, update it to a recursive pattern (e.g. `glob("docs/**/*.txt", recursive=True)`) so all 16 documents get indexed.

## Example Queries

The following example questions demonstrate the types of queries the assistant can answer:

- What is the SOP for fault code E-301 (stator overheating)?
- What are the hydraulic limits for the pitch control system (E-101)?
- Why would fault code E-501 (gearbox vibration) trigger, and what should I check first?
- What is the 8-step LOTO isolation procedure?
- How do I troubleshoot a grid overvoltage fault (F-80)?
- What causes an F-133 MPPT tracking error and how do I fix it?
- What HV PPE is required before working on a combiner box?
- What is the arc flash boundary for this equipment?
- What are the requirements for confined space entry at the tower base?
- What fall protection is required for nacelle roof access?
